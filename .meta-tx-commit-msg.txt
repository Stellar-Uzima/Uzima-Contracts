fix(meta-tx): clear CI Code-Quality failures (#834)

Follow-up to the clean-up commit. Code Quality CI was failing with:

1) clippy::deprecated_semver in erc2771_context.rs:
      #[deprecated(since = "issue-834", ...)]
   Rustlint rejects non-semver strings. Switched to the workspace
   package version (0.1.0):
      #[deprecated(since = "0.1.0", ...)]

2) clippy::too_many_arguments in test.rs:
   build_add_request(env, user, target, nonce, deadline, a, b)
   exceeded the default cap of 6. Added `#[allow(clippy::too_many_arguments)]`
   with a justification comment (every arg is a distinct named input) and
   also fixed the now-stale doc comment ("echo_add_with_sender" -> "echo_add").

3) rustfmt.toml was emitting nightly-only / deprecated-option warnings
   during cargo fmt --all -- --check. Removed the offending options:
   fn_args_layout, fn_return_indent, match_expression_indent,
   where_layout, imports_layout, format_strings, format_macro_matchers,
   format_macro_bodies, control_brace_style, match_arm_blocks,
   match_block_trailing_comma, error_on_line_overflow.
   The remaining options (hard_tabs, tab_spaces, newline_style,
   use_small_heuristics, imports_granularity, group_imports,
   reorder_imports, reorder_modules, reorder_impl_items, wrap_comments,
   comment_width, normalize_comments, max_width, edition) are stable.

   group_imports is "StdExternalCrate" (string form) instead of `true`
   because rustfmt 1.7x on stable still requires the string form;
   the bool form was rejected with `invalid type: boolean true,
   expected string` on this toolchain.

Verification locally:
  cargo fmt --all -- --check        -> 0
  cargo clippy -p meta_tx_forwarder --all-targets -- -D warnings
                                    -> 0
  cargo test --lib -p meta_tx_forwarder -> 17/17 passed

No contract behaviour changes; the deprecated alias is still wired
through to has_trusted_forwarder, the exec path is identical, and tests
still cover replay/expired/cross-forwarder/tampered-signature paths.
