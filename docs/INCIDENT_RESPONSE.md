# Incident Response Guide

This guide describes how to respond to alerts generated by the Uzima monitoring contract's telemetry system.

## Alert Sources

The monitoring contract emits structured telemetry events when thresholds are breached. The primary alert topics are:

| Topic | Threshold | Severity |
|---|---|---|
| `ERRRATE` | Error rate exceeds configured percentage | Critical |
| `GAS` | Cumulative gas exceeds configured limit | Critical |
| `STORAGE` | Storage entries exceed configured limit | Critical |

## Response Steps

### 1. Triage

When an alert fires:

- Query `get_telemetry_snapshot()` to get the current state.
- Check the `critical_count` and `security_count` fields.
- If `critical_count` is rapidly increasing, escalate immediately.

### 2. Correlate

Use the `correlation_id` from the alerting event to find related events:

- All events from the same transaction share the same `correlation_id`.
- Group events by `correlation_id` to reconstruct the execution trace.
- Look for preceding `FunctionInvoked` or `AuthFailure` events.

### 3. Diagnose

| Symptom | Likely Cause | Action |
|---|---|---|
| `ERRRATE` alert + high `error_count` | Failing function calls | Check contract logs; verify inputs |
| `GAS` alert + high `total_gas_used` | Expensive operations | Review gas per call; optimize hot paths |
| `STORAGE` alert + high `storage_entries` | Unbounded storage growth | Check for missing cleanup; archive old data |
| `AUTH_FAIL` events | Authentication failures | Verify caller addresses; check key rotation |
| `ANOMALY` events | Unusual pattern | Review correlation IDs; check for abuse |

### 4. Mitigate

Admin-only actions to reduce impact:

- **Adjust thresholds**: call `update_alert_config()` with relaxed limits to silence non-critical alerts.
- **Pause affected functions**: if a specific function is the source, coordinate with the team to pause or restrict it.
- **Escalate**: if the issue is data-loss or security-related, follow the escalation path.

### 5. Post-Incident

After resolution:

1. Document the incident with timeline and root cause.
2. Review whether threshold values should be permanently adjusted.
3. Consider adding new `TelemetryEventType` variants for similar future events.
4. Update `SCHEMA_VERSION_MINOR` if new event types are added.

## Escalation Path

1. **P3 (Low)**: non-critical threshold — investigate within 24 hours.
2. **P2 (Medium)**: repeated threshold breaches — investigate within 4 hours.
3. **P1 (High)**: `Critical` events or security anomalies — investigate within 1 hour.
4. **P0 (Urgent)**: data loss, unauthorized access, or contract compromise — immediate escalation.

## Example: Responding to an Error Rate Alert

```
1. Alert fires: THRESHOLD (severity=Critical, detail=error_rate)
2. Query get_telemetry_snapshot() → error_count=47, total_events=50
3. Error rate = 94% — this is a systemic failure
4. Find correlation_ids from the last 5 minutes
5. Trace events: FN_INVOKE → AUTH_FAIL → FN_DONE (error)
6. Root cause: callers using expired authentication tokens
7. Mitigate: notify users; adjust auth rotation policy
8. Post-incident: document in runbook; add ANOMALY event type for pattern detection
```

## Dashboard Integration

Off-consumers can parse `TelemetryEvent` data from the Soroban event stream:

- Filter by topic `(TEL, <type_symbol>)` to capture all telemetry events.
- Parse the `TelemetryEvent` struct from the event data.
- Group by `correlation_id` for trace reconstruction.
- Use `schema_version` to determine how to interpret the event fields.
